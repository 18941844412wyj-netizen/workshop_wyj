## TC-001 — 注册 + Onboarding 选角色自动应用权重
**对应 AC**：AC-001  
**前置条件**：数据库已迁移（`npm run db:migrate`），应用可访问。  
**步骤**：
1. 访问 `/register`，填写邮箱+密码，点击「注册」。
2. 注册成功后自动跳转 `/onboarding`，选择角色（如「产品经理」），点击「下一步」。
3. 调用 `GET /api/profile`，验证返回的 `role` 字段与所选角色一致，`tagWeights` 字段含非空权重对象。
4. 验证页面跳转至 `/targets`。

**预期结果**：
- 注册成功，Onboarding 选角色后系统自动应用该角色的默认信息标签权重并落库。
- `tagWeights` 字段中的值与 `solution.md` 规定的角色默认权重表一致。
- 页面跳转到监控目标页。

## TC-002 — 监控目标 CRUD
**对应 AC**：AC-002  
**前置条件**：已完成 Onboarding 的登录用户。  
**步骤**：
1. 访问 `/targets`，点击「新增竞品」，填入名称、URL、赛道（如「生成式 AI」）、采集方式「手动即时触发」。
2. 提交后验证目标出现在列表，且字段显示正确。
3. 点击编辑，修改名称，保存后验证列表更新。
4. 点击删除，验证目标从列表消失。
5. 重复新增目标，选择「固定时间每日定时」采集方式，提交成功。

**预期结果**：
- 支持新增、编辑、删除操作，状态实时刷新。
- 两种采集方式均可选择并保存。

## TC-003 — 手动触发采集→AI 分析→生成情报
**对应 AC**：AC-003  
**前置条件**：已有监控目标（手动即时触发），测试包 `ceshi/` 已就位。  
**步骤**：
1. 在 `/targets` 页面，对目标点击「立即检测」。
2. 等待分析完成（约 10～30 秒）。
3. 访问 `/inbox`，验证至少出现 1 条新情报。

**预期结果**：
- 系统执行变化检测 + AI 打标 + 分析后，至少产生 1 条情报。
- Inbox 中可见该情报。

## TC-004 — 情报字段完整性
**对应 AC**：AC-004  
**前置条件**：TC-003 已产生情报。  
**步骤**：
1. 在 Inbox 点击一条情报，进入 Inspector 详情 Tab。
2. 验证以下字段均存在且非空：变化内容（What Changed）、战略意义（Strategic Significance）、行动建议（Action Plan，含当前角色个性化 + 通用三视角兜底）、优先级（紧急/中等/低之一）、原文引用锚点（View Source 可展开）。
3. 调用 `GET /api/insights`，验证 `labels` 字段数组长度 ≥ 1，取值在六大标签范围内。

**预期结果**：
- 情报含全部五要素字段且无空值。
- 每条情报至少命中 1 个信息标签。
- 优先级为「紧急」「中等」「低」三档之一。

## TC-005 — 个性化 Inbox 排序
**对应 AC**：AC-005  
**前置条件**：已有多条情报（通过 `scripts/seed-mock-intels.ts` 造数或多次触发分析）；存在不同标签/优先级的情报。  
**步骤**：
1. 以「产品经理」角色登录，查看 Inbox 列表顺序。
2. 验证「紧急」且命中高权重标签（功能/产品定价）的情报排在前面。
3. 验证低权重标签（如「招聘」）的情报仍出现在列表（降权但未消失）。
4. 在设置中切换为「市场营销负责人」，返回 Inbox，验证排序发生变化。

**预期结果**：
- 高优先级 + 高权重领域情报置顶，低权重领域情报降权但保留（非隐藏）。
- 切换角色后排序随之变化。

## TC-006 — Inspector 详情 + 查看原文
**对应 AC**：AC-006  
**前置条件**：Inbox 中存在情报。  
**步骤**：
1. 在 Inbox 点击任意情报，右栏进入 Inspector 详情 Tab。
2. 验证展示：变化内容、战略意义、行动建议（含当前角色建议 + 通用视角）、原文引用锚点。
3. 点击「查看原文」，验证能展开变化对比面板，显示变化前后的文本对比（或引用段落）。

**预期结果**：
- Inspector 详情 Tab 展示全部分析字段。
- 原文来源可展开，不仅是标签占位。

## TC-007 — 引用式深度对话（资料不足兜底）
**对应 AC**：AC-007  
**前置条件**：Inbox 中存在情报，Inspector 详情 Tab 可用。  
**步骤**：
1. 切换到 Inspector 深度对话 Tab，选中一条情报（或多条）作为引用。
2. 输入与情报内容相关的问题，点击发送，验证 AI 给出基于情报内容的回答。
3. 输入与引用情报完全无关的问题（如「明天天气如何？」），验证回答明确说明「资料不足」或类似表达，不臆造答案。

**预期结果**：
- 在引用范围内的问题获得有依据的回答。
- 超出范围的问题回答明确包含「资料不足」或「依据不足」字样，不编造信息。

## TC-008 — 多会话管理与历史持久化

**前置条件**：深度对话 Tab 可用。

**预期结果**：

## TC-009 — 情报反馈打标 + 核心池
**对应 AC**：AC-009  
**前置条件**：Inbox 中存在情报，Inspector 可用。  
**步骤**：
1. 对一条情报，在 Inspector 底部打开反馈面板，选择「优先级错误」标签，填写问题模块（如「AI 分析」）及补充说明，提交。
2. 刷新页面，验证反馈状态仍显示（未被清空）。
3. 对另一条情报，反馈标记「有用」，提交后切换到核心池视图，验证该情报出现在核心池。

**预期结果**：
- 反馈刷新后持久存在。
- 标记「有用」的情报自动进入核心池视图。

## TC-013 — Vercel Cron 自动采集（UR-3）
**对应 AC**：AC-013  
**前置条件**：已配置 `CRON_SECRET` 环境变量，存在绑定「固定时间每日定时」采集方式的监控目标。  
**步骤**：
1. 直接调用 Cron 触发端点（`GET /api/cron/analyze?secret=<CRON_SECRET>`），模拟定时触发。
2. 等待响应，验证返回成功状态。
3. 刷新 Inbox，验证有新情报产生，字段完整（五要素 + 信息标签 + 优先级）。

**预期结果**：
- 无人工介入的情况下，Cron 端点可完整跑通「检测→打标→分析」全链路并产出新情报。

## TC-014 — 情报归档状态持久化（UR-4）
**对应 AC**：AC-014  
**前置条件**：Inbox 中有情报。  
**步骤**：

## TC-015 — 邮件主动通知（UR-6）
**对应 AC**：AC-015  
**前置条件**：设置中已配置推送邮箱，`RESEND_API_KEY` 已配置，存在「紧急」优先级情报。  
**步骤**：
1. 进入设置→邮件通知，添加推送邮箱，开启通知，保存。
2. 调用 `POST /api/notify`（或等待 Cron 产生紧急情报），触发邮件通知。
3. 验证配置的推送邮箱收到含情报摘要的邮件（含标题/战略意义/行动建议/链接）。
4. 对同一情报再次调用通知接口，验证不重复发送（去重生效）。
5. 在设置中关闭邮件通知，再次触发，验证不再收到邮件。

**预期结果**：
- 紧急/高匹配情报触发邮件推送；同一情报不重复发送；关闭后不再收到。
