## 1. 结论摘要（先给结论）

- 一句话目标：为跨境卖家（Linkfox AI 视角）提供一个 Web 版竞品情报监控代理 MVP，把「网页变化」自动解读成「变化 → 战略意图 → 分角色行动建议」的结构化情报，并围绕**「因人而异地推送 + 深度对话式挖掘」**这一核心价值，支持对单条情报的 AI 深度追问。
- 三大核心亮点（本次重点突出，基本功能不变）：
  1. **信息收集层（因人而异的入口）**：注册后以表单收集「我的角色 + 我更关注的信息」，为不同产品/岗位建立**角色标签**与**信息标签**（定价 / 功能 / 更新日志 / 招聘 / 营销活动 / 合规条款），并支持自定义信息权重（多看哪些、少看哪些）；有重点、有取舍，但不完全屏蔽其它领域。后续支持随时更改角色与权重。
  2. **信息匹配层（AI 定义价值 + 精准匹配）**：由 AI 对网页变化按**六大信息标签**打标，再按用户的角色与权重做「人—信息」匹配（产品经理 / 市场营销负责人 / 创业者·创始人 / 投资人各有默认关注偏好），支持自定义扩展更多角色。
  3. **信息消费层（引用式深度对话）**：在情报详情内以「引用信息 → 对话信息」的方式二次萃取价值，借助 AI 分析能力多轮深挖，会话持续落盘直到用户主动终止。
- 本次 In / Out 边界：In = 带「角色+信息标签」的最简登录/Onboarding + 监控目标管理 + **定时自动采集与解读（无需人工介入，P0）** + **采集层「markdown 抓取 + JS 注入降级（markdown<3 行触发）」能力（本期默认输入=测试包，对齐原始《生成MD测试用例》）** + 变化检测 + AI 维度定义与信息打标 + 真实大模型分析 + 按角色/权重个性化的情报 Inbox（含情报状态：未读/已读/归档）+ 引用式多轮深度对话 + **主动通知（邮件推送重点/紧急情报，P1）** + 情报反馈；Out = 真实全站规模化爬取与反爬对抗（采集能力已设计，真实抓取可行性见 V-010）、行业政策源采集、竞品自动推荐（UR-2 P2）、Slack/IM 等其它推送渠道（邮件已纳入本期）、多用户权限、质量看板、社媒/流量/AI 提及度信号、跨历史情报追问。
- 推荐方案：**测试包驱动的情报 MVP（Test-Pack-Driven MVP）** —— 以 `index.html` 为基准、`cases/` 变体模拟网页变化，经变化检测后交由真实大模型定义价值维度、打信息标签并生成分角色结构化情报，按用户角色与权重个性化分发，部署于 Vercel。
- 优先验证点：`V-001`（Vercel 兼容的数据库选型）、`V-002`（3 天工时能否覆盖全部 MVP 功能）、`V-003`（大模型密钥/成本可用性）、`V-007`（角色×信息标签匹配与打标可用性）。

## 2. 推荐方案（唯一）

- 方案名：**测试包驱动的情报 MVP（Test-Pack-Driven MVP）**
- 主流程 / 关键机制（按三大核心亮点分层组织；未标注的即为不变的基本功能）：
  1. **登录 + Onboarding〔信息收集层〕**：最简账号密码注册/登录 + 会话保持；注册后选择**角色标签**（内置四类 + 支持自定义扩展），系统**自动应用该角色默认信息标签权重**；权重可在设置页按角色分别调整，Inbox 支持角色快切预览不同视角行动建议。
  2. **监控目标管理**：手动登记竞品（名称 + URL + 赛道：生图/生视频/Agent），提供 URL 后系统自动抓取（提取 markdown，markdown<3 行时 JS 注入降级再抓）并识别变化；采集方式支持「手动即时触发」与「固定时间采集（每日定时）」。本期默认输入为测试包（真实抓取可行性见 V-010）。
  3. **定时自动采集与解读〔UR-3 · P0〕**：由 **Vercel Cron 定时**对已绑定的监控目标自动执行「采集 → 变化检测 → 打标 → 分析」全链路，无需人工点击；同时保留「手动触发分析」按钮作为补充。
  4. **变化检测**：将「基准版本」与「变体版本」做文本 diff，过滤纯样式/噪音，输出「有意义的变化候选」。
  5. **AI 价值定义与打标〔信息匹配层〕**：对每条变化候选按**六大信息标签**（见 2.1）判定归属并打标；A/B 测试、纯样式变化、幻觉诱饵等噪音不生成情报（见 2.1 过滤项）。
  6. **AI 分析（真实大模型）**：基于「变化候选 + 原文内容」生成结构化情报：变化内容 / 战略意义（战略意图）/ 行动建议（按用户角色个性化 + 通用三视角销售·产品·营销兜底）/ 优先级三档（紧急 > 中等 > 低）/ 原文引用锚点。
  7. **个性化情报 Inbox〔信息匹配层落地〕**：**双栏工作区**——左栏个性化排序列表（晨报/核心池/全部三视图 + 筛选 + 角色快切），右栏 Inspector 承载详情与深度对话；情报状态 + 核心信息池。
  8. **主动通知〔UR-6 · P1〕**：邮件主动推送，支持**多个推送邮箱**；设置中独立配置推送时间、推送内容。
  9. **引用式深度对话〔信息消费层〕**：在 Inbox Inspector 深度对话 Tab 以「引用一条/多条情报 → 就它对话」的方式二次萃取价值，支持多会话管理与跨情报联合分析；`/chat` 为兼容重定向路由。
  10. **反馈**：对每条情报打标（有用 / 幻觉 / **漏抓** / 优先级错误 / 建议废话 / **A/B 测试误报** / 其他），支持**问题模块定位**与补充说明（对齐原始《效果评估标准》Bug 反馈规范的问题类型）；标记「有用」自动加入核心信息池。
- 关键边界/取舍：
  1. 采集层按「真实站点 markdown 抓取 + JS 注入降级（<3 行触发）」设计（对齐原始《生成MD测试用例》），但**本期默认输入为受控测试包**，换取 3 天内稳定可演示；真实站点抓取的 Serverless 可行性以 V-010 早期验证（`raw.md` 第 2/6 轮）。
  2. 大模型真实接入，但用可配置接入层封装（模型/密钥/超时可配），保证核心价值又便于替换（`raw.md` 第 4 轮）。
  3. 认证只做基础注册/登录，不做多租户与权限矩阵（`raw.md` 第 3 轮）。
  4. Vercel Serverless 部署，存储用 Serverless 友好托管 DB（非本地 SQLite）；采集/分析以 **Vercel Cron 定时自动触发为主（P0，满足 UR-3「无需人工介入」）**，手动触发作为补充（`raw.md` 第 5 轮）。
  5. 主动通知本期只做**邮件**推送（满足 UR-6 P1），需接入一个邮件发送服务（如 Resend / SMTP）；Slack/IM 等渠道列为后续演进。
- 为什么选它：
  1. 直接命中 `raw.md` 的「3 天 MVP + 前后端分离 + 可部署 + 浏览器访问」硬约束。
  2. 用测试包规避真实爬取的不确定性，把有限工时投在「AI 解读 + 因人而异分发 + 深度对话」这三个核心价值上。
  3. 保留真实爬取、政策源、追问深挖等扩展点，后续演进平滑。

### 2.1 三大核心亮点（本次重点突出）

> 设计动机：对齐 Linkfox「功能模块子 Agent」的运转模式——不同产品/岗位关注的信息不同。目标是**提升信息准确性、减少信息获取者的精力消耗**，只把「最关注、最有价值」的信息重点推送给对的人；同时保持通用性，不完全屏蔽其它领域，做到**有重点、有取舍**。

#### 亮点 1 · 信息收集层（因人而异的入口）

- **角色标签**：注册后表单收集「我的角色是 __」「我更关注的信息是 __」，为该用户建立角色画像。首批角色：产品经理、市场营销负责人、创业者/创始人、投资人；架构上支持后续迭代扩展更多角色（如技术人员、运营人员）。
- **信息标签 + 权重**：信息标签覆盖网页变化的**六大真信号维度**（见下表）；支持用户**自定义配置信息权重**（多看哪些、想看哪些）。例如市场负责人可将「定价」「营销活动」权重调高，优先推送相关情报，但其它标签仍以低权重保留。
- **六大信息标签（经 `ceshi/cases/` 全量 case 验证）**：

| 信息标签 | 抓什么 | 测试 case |
|---|---|---|
| 定价 | 价格数字、套餐、credits 额度（含价不变、量缩水的变相涨价） | Z-Pricing-1/2 |
| 功能 | 新增/下线功能、能力边界 | Z-Feature-1/2 |
| 更新日志 | 版本号、发布日期、变更项（含偷改历史日期） | Z-Release-1/2 |
| 招聘 | 新岗位、地点、招哪个团队 | Z-Hiring-1/2 |
| 营销活动 | 限时活动、折扣码、优惠力度 | Z-Marketing-1/2 |
| 合规条款 | 数据保留期、权益有效期、授权/隐私条款 | Z-Recall-1/2 |

- **噪音过滤（识别但不报）**：A/B 测试（Z-AB-1/2）、纯样式变化（Z-Noise-1/2）、幻觉诱饵（Z-Hallucination-1/2，如客户数/媒体背书/未上线预告）不生成情报。
- **可变更**：角色与权重支持在设置中随时更改，画像即时生效于后续分发。
- **实现策略**：Onboarding 表单落库为用户画像（角色 + 信息标签权重向量），作为信息匹配层的输入。

#### 亮点 2 · 信息匹配层（AI 定义价值 + 精准匹配）

- **AI 打标**：对每条有意义的变化候选打上**一个或多个信息标签**（六大标签之一），并产出优先级与原文锚点。
- **人—信息匹配**：将 AI 产出的信息标签与用户画像（角色 + 权重）做匹配打分，决定情报的个性化排序与是否重点推送；角色选择可提供各标签的默认权重建议。
- **可自定义**：标签体系与角色可扩展（技术人员、运营人员等都是产品重要环节，后续迭代纳入）。
- **分角色默认关注偏好（首批 4 角色，用于 Onboarding 默认权重建议）**：

| 角色 | 默认高关注信息标签 |
|---|---|
| 产品经理 | 功能、更新日志 |
| 市场营销负责人 | 定价、营销活动 |
| 创业者/创始人 | 功能、定价、招聘 |
| 投资人 | 定价、招聘、合规条款 |

#### 亮点 3 · 信息消费层（Inspector 引用式深度对话）

- **二次萃取**：在 Inbox 右栏 Inspector 深度对话 Tab，用户可**引用一条或多条情报**发起对话，支持跨情报联合分析。
- **多会话管理**：支持创建/切换/终止多个对话会话，历史持续落盘。
- **多轮深挖**：对话为多轮，基于「被引用情报 + 引用范围 + 会话上下文」作答；资料不足时坦白，不臆造、不联网现抓。

## 3. 备选方案（差异明显）

### 3.1 备选方案：纯前端演示（No-Backend Demo）

- 核心机制：不做后端与数据库，前端内置静态数据与预生成情报，纯展示交互。
- 主流程：加载静态情报 JSON → 渲染 Inbox → 假的追问（预置回答）。
- 边界与取舍：最快出成品；但无真实登录、无持久化、AI 非真实。
- 适用前提：仅需一个「看起来能用」的界面 Demo。
- 不选原因：违背 `raw.md`「前后端分离 + 注册登录 + 可部署」的明确要求，且丢失核心价值（真实 AI 解读）。

### 3.2 备选方案：真实线上爬取 MVP（Live-Crawl MVP）

- 核心机制：真实抓取少量/多量线上竞品站点，markdown 提取 + JS 注入降级，再做变化检测与分析。
- 主流程：定时抓取线上页面 → 存快照 → diff → AI 分析。
- 边界与取舍：最贴近最终形态；但反爬/JS 渲染/合规/稳定性风险高。
- 适用前提：有充足时间与稳定抓取基础设施。
- 不选原因：3 天内难以稳定，已在 `raw.md` 第 2 轮裁决为「后续演进」。

### 3.3 备选方案：Mock AI MVP（模拟分析）

- 核心机制：全流程真实，但 AI 分析用预置文案/规则模板占位，不接真实模型。
- 主流程：变化检测 → 模板生成情报 → Inbox。
- 边界与取舍：省去模型接入成本与不确定性；但产出无真实洞察，等同高级 RSS。
- 适用前提：模型密钥/预算完全不可得时的临时降级。
- 不选原因：丢失「像分析师一样思考」的核心竞争力，已在 `raw.md` 第 4 轮裁决为真实接入。

## 5. 验证清单（V-xxx，可执行）

- V-001 Vercel 兼容的数据库选型未定
  - 风险/假设：Vercel Serverless 文件系统临时化，本地 SQLite 无法持久化，若沿用会导致数据丢失。
  - 方法：在 Vercel 上试接一个 Serverless 友好托管 DB（Vercel Postgres 或 Neon），跑通「注册用户 + 存一条情报 + 重启后仍在」。
  - 成功/失败信号：重启/重新部署后数据仍存在为成立；否则不成立。
  - Owner：DEV
  - 截止：开发启动后 0.5 天内
  - 触发动作：不成立则改用 Neon/Supabase 等托管 Postgres，并更新数据访问层。

- V-002 3 天工时能否覆盖全部 MVP 功能（含 S9 追问）
  - 风险/假设：登录 + 监控管理 + 变化检测 + 真实 AI + Inbox + S9 追问 + 反馈，功能偏多，3 天可能超期。
  - 方法：开工前做半天工时切分，按「核心闭环（登录→检测→AI→Inbox→反馈）」优先、S9 追问列为最后一个可裁剪项排期。
  - 成功/失败信号：核心闭环 Day2 结束前可演示为成立；否则不成立。
  - Owner：DEV + PM
  - 截止：开发启动当天
  - 触发动作：不成立则将 S9 追问降级为「后续演进」，先保核心闭环上线。

- V-003 大模型密钥/模型可用性与成本
  - 风险/假设：真实 LLM 调用需要密钥与预算，且三角色建议使单条分析调用量偏大。
  - 方法：确认可用模型与密钥；对单条情报做一次端到端调用，记录耗时与 token 成本。
  - 成功/失败信号：单条分析可稳定返回且成本可接受为成立；否则不成立。
  - Owner：PM 提供密钥 / DEV 接入
  - 截止：开发启动后 0.5 天内
  - 触发动作：不成立则低优先级情报只生成通用建议、或改用更低成本模型。

- V-004 变化检测去噪效果（纯样式/A-B 误报）
  - 风险/假设：diff 可能把无意义的样式变化误报为「有意义变化」。
  - 方法：用 `cases/` 中「仅样式改动」与「真实内容改动」两类页面各测若干，统计误报。
  - 成功/失败信号：纯样式改动不产生情报、内容改动能被捕获为成立；否则不成立。
  - Owner：DEV
  - 截止：Day2 结束前
  - 触发动作：不成立则加入正文抽取/文本归一化预处理再 diff。

- V-005 项目知识库缺失（Context Gap）
  - 风险/假设：`product.md`/`glossary.md`/`products/index.md`/`components/index.md` 均缺失，Impact Analysis 无权威输入。
  - 方法：确认是否为全新项目（greenfield）；如是，则本 spec 内定义模块即为新增基线。
  - 成功/失败信号：确认 greenfield 且模块清单被采纳为成立；否则需先补项目知识库。
  - Owner：PM
  - 截止：方案评审时
  - 触发动作：非 greenfield 则先执行 project-discover 补齐权威入口再复核 Impact Analysis。

- V-006 AI 事实性/幻觉基线
  - 风险/假设：真实模型可能产出原文没有的事实（幻觉），损害可信度。
  - 方法：抽查若干条情报，逐字核对 What Changed 与原文；同时校验「答不上来是否坦白」。
  - 成功/失败信号：抽查无编造事实、资料不足时能坦白为成立；否则不成立。
  - Owner：PM（验收）/ DEV（Prompt 调优）
  - 截止：MVP 演示前
  - 触发动作：不成立则强化「仅基于原文作答 + 强制引用锚点」的提示约束。

- V-007 角色×维度匹配与信息打标的可用性（三大亮点核心）
  - 风险/假设：AI 对六大信息标签的打标可能不稳定，个性化排序可能与用户直觉不符，导致「因人而异分发」失效。
  - 方法：用 `cases/` 变体覆盖 4 角色的典型标签（如定价变动→市场营销、新功能上线→产品经理），核对情报被打上的信息标签是否正确；再切换不同用户画像（角色+权重）核对 Inbox 排序是否随之变化且重点领域置顶。
  - 成功/失败信号：信息标签命中正确、切换画像后排序有差异且重点领域优先为成立；否则不成立。
  - Owner：PM（验收）/ DEV（Prompt 与打分调优）
  - 截止：Day2 结束前
  - 触发动作：不成立则将维度体系收敛为「基于关键词的规则打标 + AI 辅助」，或将个性化排序降级为「筛选+高亮」而非重排。

- V-008 邮件发送服务可用性（UR-6 主动通知 P1）
  - 风险/假设：邮件推送需第三方发送服务（Resend/SMTP）+ 密钥，可能存在送达率、配额或配置问题。
  - 方法：接入发送服务，对一条「紧急/高匹配」情报触发一次真实邮件发送，确认收件箱可收到且含情报摘要与详情链接。
  - 成功/失败信号：能稳定发出且被收到为成立；否则不成立。
  - Owner：PM 提供服务/密钥 / DEV 接入
  - 截止：开发启动后 1 天内
  - 触发动作：不成立则降级为「站内通知/Inbox 红点」保底满足"主动提醒"，邮件转后续补齐。

- V-010 真实站点采集能力可行性（markdown 抓取 + JS 注入降级，对齐原始《生成MD测试用例》）
  - 风险/假设：真实站点抓取需 `fetch→markdown 提取→(markdown<3 行则 JS 注入渲染再抓)`，JS 注入依赖无头浏览器/渲染服务，在 Vercel Serverless 上偏重、可能冷启动/超时（300s）。
  - 方法：选 1–2 个真实站点（含需 JS 注入的 klingai/ihuiwa 类）走完整链路，记录耗时/成功率；确认 <3 行触发降级逻辑生效。
  - 成功/失败信号：链路跑通且不超时为成立；否则不成立。
  - Owner：DEV
  - 截止：Day2 结束前
  - 触发动作：不成立则本期降级「仅静态 fetch + 测试包默认输入」，真实 JS 注入抓取转后续演进（不阻塞核心闭环）。

## 8. Mini-PRD（本 spec 暂走快速通道，跳过独立 prd.md 时必填）

- **MVP 范围**：
  - In：
    - 最简注册/登录 + 会话保持（单一账号）。
    - 〔亮点1〕Onboarding 用户画像：注册后收集角色标签（PM/市场/创始人/投资人，可扩展）+ 信息标签权重（定价/功能/更新日志/招聘/营销活动/合规条款），支持后续更改。
    - 监控目标管理：登记竞品（名称 + URL + 赛道），设置采集方式（手动即时触发/固定时间每日定时），系统自动监控 URL 变化（采集层：markdown 抓取 + JS 注入降级，本期默认测试包输入，真实抓取见 V-010）。
    - 〔UR-3 · P0〕定时自动采集与解读：Vercel Cron 定时自动跑「检测→打标→分析」，无需人工；手动触发补充。
    - 变化检测：基准 vs 变体文本 diff，过滤纯样式噪音，输出变化候选。
    - 〔亮点2〕AI 价值定义与信息打标：按六大信息标签为变化候选打标；过滤 A/B/样式/幻觉噪音。
    - AI 分析：真实大模型产出 What Changed / Why it matters / Action Plan（按用户角色个性化 + 通用三视角兜底）/ 优先级三档 / 原文锚点。
    - 〔亮点2落地〕个性化情报 Inbox：结合用户角色/权重的个性化排序（重点置顶、其它领域降权保留不屏蔽）+ 列表（优先级排序、紧急置顶）+ 情报状态（未读/已读/归档，UR-4）+ 详情（完整分析 + View Source）+ 筛选（赛道/信息标签/优先级）+ 晨报视图。
    - 〔UR-6 · P1〕主动通知：对紧急/高匹配情报邮件推送（去重+频控，设置可开关）。
    - 〔亮点3〕全局引用式深度对话：在独立 /chat 页面引用任意情报发起多轮对话；资料不足坦白；全局会话持续落盘直到用户主动终止。
    - 反馈：情报打标并落库。
  - Out（后续演进）：真实线上规模化爬取的稳定性/反爬对抗（采集 markdown+JS 注入能力已入本期、默认测试包输入，见 V-010）；行业政策源采集；竞品自动推荐（UR-2 P2）；Slack/IM 等其它推送渠道（邮件已纳入本期）；多用户权限与数据隔离；质量看板与自动回归；社媒/流量/AI 提及度信号；跨历史情报追问；反馈反哺个性化权重；更多角色（技术/运营等）；周报/月报。
  - 效果评估对标基线（对齐原始《效果评估标准》，用于验证阶段而非产品功能）：**Visualping 免费版**作为抓取覆盖率基准（确认网页是否真变化，排查漏抓 vs 判噪）；**Perplexity Pro / GPT-5.5** 作为洞察质量基准（同源对比分析深度）；**噪音率（Noise Rate）** 作为日常巡检质量指标。
- **验收标准（AC）**：
  1. 用户可注册并登录；注册后可在 Onboarding 选择角色并配置信息标签权重，登录后进入个性化 Inbox；角色/权重可在设置中更改。
  2. 用户可登记至少 1 个竞品并绑定测试包页面。
  3. 触发分析后，系统能对「基准 vs 变体」产出至少 1 条结构化情报，且每条被打上 ≥ 1 个信息标签。
  4. 每条情报包含：What Changed、Why it matters、Action Plan（含面向当前用户角色的个性化建议）、优先级（三档之一）、可点击 View Source 查看原文来源。
  5. 纯样式改动的变体不产生「重大变化」情报（去噪生效）。
  6. Inbox 排序随用户角色/权重变化：重点领域情报被优先置顶，其它领域降权保留而非消失。
  7. 〔UR-3〕Vercel Cron 定时任务可在无人工介入下自动跑通「检测→打标→分析」并产出新情报。
  8. 〔UR-4〕用户可将情报标记为已读/归档，状态持久化。
  9. 〔UR-6〕新产生的紧急/高匹配情报会通过邮件推送给对应用户；用户可在设置中开关通知。
  10. 用户可在情报详情引用该情报/原文片段发起多轮深度对话，回答基于当前情报与被引用原文；原文无答案时明确告知「资料不足」；会话历史落盘，直到用户终止。
  11. 用户可对情报打标反馈（标签集：有用 / 幻觉 / 漏抓 / 优先级错误 / 建议废话 / A/B 测试误报 / 其他 + 问题模块 + 补充说明），刷新后反馈仍在（持久化生效）。
  12. 应用部署在 Vercel，浏览器可访问，数据重新部署后不丢失。
- **交互变化结论**：有但简单——为全新 MVP，页面为登录/注册、Onboarding 画像、监控目标管理、个性化 Inbox 列表、情报详情（含引用式深度对话与反馈）、设置（改角色/权重）；交互清晰，可直接进入原型/开发。
- **影响面**：
  - 页面/入口：`/login`、`/register`、`/onboarding`（选角色自动配权重）、`/targets`（监控目标+采集方式+编辑）、`/inbox`（双栏工作区，含 Inspector）、`/inbox/:id` 与 `/chat`（兼容重定向）、`/settings`（角色与权重/邮件通知两 Tab）。
  - 接口（Serverless Functions，示意）：`POST /api/auth/register`、`POST /api/auth/login`、`GET/PUT /api/profile`（角色+权重+通知偏好）、`GET/POST /api/targets`、`POST /api/analyze`（含价值定义+打标）、`GET /api/cron/analyze`（Vercel Cron 定时自动采集与解读，UR-3）、`GET /api/insights`（按画像个性化排序）、`GET /api/insights/:id`、`PATCH /api/insights/:id`（已读/归档状态，UR-4）、`POST /api/insights/:id/chat`（引用式多轮对话）、`POST /api/insights/:id/feedback`、`POST /api/notify`（邮件推送，UR-6，供 Cron/分析后调用）。
  - 权限点：MVP 仅「已登录」这一道门槛，无细粒度权限；`/api/cron/*` 用 Vercel Cron 密钥保护。
