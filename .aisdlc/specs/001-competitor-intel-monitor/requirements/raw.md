我要做一个自动化的产品竞争情报监控代理 Web 应用。
用户：产品经理、市场营销负责人、创始人、投资人。
核心功能：注册登录功能、监控竞争产品功能、自动地采集与解读功能、行动建议功能、主动通知功能、产品反馈与改进功能
要求：前后端分离，能部署上线，支持浏览器访问。
要求：3天内完成 MVP。

## 澄清记录

### 第 1 轮（范围裁决）
- 本轮结论（可直接引用到 solution）：本次 spec 以「3 天可交付的 MVP」为准，先交付能跑通的最小闭环；完整愿景（行业政策源、Agent 追问深挖、质量看板、社媒/流量/AI 提及度信号、外部推送渠道等）列为后续演进。
- 本轮新增/更新的约束：
  1. solution.md 的推荐方案必须锚定「3 天 MVP」，避免把大愿景当作本期承诺。
  2. 完整愿景内容保留为「后续演进」章节，不进入本期验收口径。
- 关键决策：范围 → 选择「3 天 MVP 优先」。
- 遗留歧义：注册登录 / AI 接入 / 存储部署 仍待澄清（见后续轮次）。

### 第 2 轮（采集数据来源）
- 本轮结论（可直接引用到 solution）：MVP 采集数据来源使用「提供的测试包」——以 `index.html` 为基准原版、`cases/` 目录下的变体页面覆盖 `index.html` 来模拟网页变化，用于变化检测与 AI 分析演示；不在本期做真实线上多站点爬取。
- 本轮新增/更新的约束：
  1. 采集引擎在 MVP 阶段以「读取本地/受控 HTML」为输入，保留未来接入真实爬取（含 JS 注入降级）的扩展点。
  2. 「Visualping 覆盖率对标 / 真实全站爬取」列为后续演进。
- 关键决策：采集方式 → 选择「测试包受控演示」。
- 遗留歧义：AI 接入 / 存储部署 仍待澄清。

### 第 3 轮（注册登录范围）
- 本轮结论（可直接引用到 solution）：MVP 做「最简单的账号密码登录」——支持注册与登录，单一用户角色，不做邮箱验证、找回密码、第三方登录、精细权限；用于满足 raw.md 的登录功能并隔离基本会话。
- 本轮新增/更新的约束：
  1. 认证仅覆盖基础注册/登录 + 会话保持；不做多租户数据隔离与角色权限矩阵。
  2. 「完整多用户 / 权限 / 数据隔离」列为后续演进。
- 关键决策：注册登录 → 选择「最简账号密码登录」。
- 遗留歧义：存储部署 仍待澄清。

### 第 4 轮（AI 分析落地形态）
- 本轮结论（可直接引用到 solution）：MVP 真实接入大模型 API 产出洞察（变化→意图→建议、以及 S9 情报追问），并以可配置的接入层封装（模型/密钥/超时可配），保证「事实性 + 溯源 + 答不上来坦白」的质量基线。
- 本轮新增/更新的约束：
  1. AI 输出必须基于「变化检测结果 + 原文内容」作答，禁止臆造；追问基于「当前情报 + 原文来源」。
  2. 需要一个可配置的模型接入层（便于切换模型/填入密钥）。
- 关键决策：AI 接入 → 选择「真实接入大模型 API」。
- 遗留歧义：数据库选型（Vercel 下需 Serverless 友好 DB）转入 solution 验证清单。

### 第 5 轮（存储与部署形态）
- 本轮结论（可直接引用到 solution）：MVP 部署到 Vercel。
- 本轮新增/更新的约束（技术含义）：
  1. Vercel 为 Serverless 平台，本地 SQLite 不可持久化 → 存储采用 Serverless 友好的托管数据库（推荐 Vercel Postgres / Neon Postgres）。
  2. 后端以 Serverless Functions 形态提供 API；前端可用 Next.js 或独立前端 + Serverless API（保持前后端分离）。
  3. 周期采集用 Vercel Cron 或手动触发（MVP 使用测试包，可先提供「手动触发分析」按钮，Cron 作为可选）。
- 关键决策：部署 → 选择「Vercel」。
- 遗留歧义：数据库具体选型作为验证项在 solution 中确认（不阻塞方案）。

### 第 6 轮（进入实施计划前·原始需求对齐复核）
- 触发：进入 D2→I 阶段前，比对 `yuanshixuqiu/` 三份原始文档（原始需求 / 效果评估标准 / 生成MD测试用例）与当前 Spec 的偏差，经用户逐项裁决。
- 本轮结论（可直接引用到 solution/design）：
  1. **A3 采集能力对齐（覆盖第 2 轮的部分口径）**：采集层按原始《生成MD测试用例》升级为「真实站点 markdown 抓取 + JS 注入降级（markdown < 3 行时触发 JS 注入渲染再抓）」并写入 SSOT/ADR-0008；**本期默认输入仍为受控测试包**，真实站点抓取的 Serverless 可行性列为验证项 R-010/V-010（不阻塞 3 天核心闭环）。落地方式=「能力入案 + 测试包默认 + R-010 验证」。
  2. **C 类效果评估对齐（对齐原始《效果评估标准》）**：①新增效果评估对标基线——Visualping（抓取覆盖率）、Perplexity Pro/GPT-5.5（洞察质量）、噪音率（Noise Rate）质量指标，用于验证/测试阶段；②反馈标签补齐为「有用/幻觉/漏抓/优先级错误/建议废话/A-B 测试误报/其他」（新增「漏抓」「A/B 测试误报」），并同步 demo（`FeedbackTag` 与 Inbox 反馈 UI）。
- 维持现状（经确认不改）：A1 多维度信号（社媒/流量/AI 提及度，仅网站）、A2 Slack/IM 渠道（仅邮件）、B 三大亮点（角色权重个性化 + 引用式深度对话 + 自定义角色）均保持本期范围不变。
- 关键决策：采集能力 → 对齐原始（能力入案 + 测试包默认 + R-010 验证）；效果评估 → 补齐对标基线与反馈标签。
- 遗留歧义：无（R-010 真实抓取可行性于开发早期验证）。
